# 2. Project Execution

## 2.1 Data Acquisition

### 2.1.1 Primary corpus

The primary dataset is **AG News**<sup>[1]</sup>, retrieved
through the Hugging Face ``datasets`` library:

```python
from datasets import load_dataset
ds = load_dataset("ag_news")
# train: 120,000 articles | test: 7,600 articles
# labels: World (0), Sports (1), Business (2), Sci/Tech (3)
```

A stratified 90 / 10 split of the official training set yields 108,000
training examples and 12,000 validation examples; the official test split
remains untouched and is used only for the final reported metrics.

### 2.1.2 Multilingual extension (Vietnamese and French)

To investigate cross-lingual transfer, two parallel copies of AG News
are generated by machine translation. Phase 5 is split into a
Vietnamese sub-phase (5A) and a French sub-phase (5B); both follow the
same OPUS-MT translate + back-translate pipeline, parameterised by the
target language code:

| Step | Tool (target = `vi` / `fr`)                                                   | Input                          | Output                       |
|------|-------------------------------------------------------------------------------|--------------------------------|------------------------------|
| 1    | ``Helsinki-NLP/opus-mt-en-{target}``                                          | English AG News (all splits)   | Target-language AG News      |
| 2    | ``Helsinki-NLP/opus-mt-{target}-en``                                          | Target-language AG News (train) | Pivot English (intermediate) |
| 3    | ``Helsinki-NLP/opus-mt-en-{target}``                                          | Pivot English (intermediate)   | Augmented target language    |

The OPUS-MT models<sup>[14]</sup> are batched on the
GPU with beam search width 4 and a maximum sequence length of 512
tokens. Back-translation<sup>[15]</sup> is applied to the
training split only; validation and test stay at the direct
translation. Translated and back-translated corpora are saved as
Hugging Face ``DatasetDict`` objects under ``data/processed/`` so any
downstream phase can re-use them through
``datasets.load_from_disk``.

### 2.1.3 Reproducibility and provenance

All datasets are versioned alongside the code through a single seed
(``training.seed = 42``) applied centrally by ``src.utils.repro.set_global_seed``.
The seed governs Python's ``random``, NumPy, PyTorch, and the stratified
split that produces the train / validation partitions. Runtime
dependencies are pinned in ``requirements.txt``; the headline numbers
reported in Section 3 were produced with ``torch==2.6.0+cu124``,
``transformers==4.45``, ``optimum==1.23``, ``onnxruntime==1.20``,
``scikit-learn==1.5``, and ``numpy==1.26``. The single-stream latency
table in Section 3.5.3 was produced on an Intel Xeon Platinum 8558
(Sapphire Rapids, AVX-512 VNNI + AMX_INT8) inside the rented cloud GPU
instance; the exact CPU model name is captured in
``outputs/evaluation/latency_pareto.meta.json`` by
``scripts/benchmark_all_latency.py:_cpu_info()``.

## 2.2 Training Methodology

The project compares three families of models on a single train / test
split, each implemented behind a uniform ``fit`` / ``predict`` /
``predict_proba`` interface so that the evaluation pipeline (Phase 7) can
treat them interchangeably.

### 2.2.1 Classical baselines (Phase 3)

- **TF-IDF + Logistic Regression**. ``TfidfVectorizer`` with bigrams,
  ``min_df=2``, ``max_df=0.95``, ``sublinear_tf=True``, followed by
  multinomial ``LogisticRegression`` (``solver="lbfgs"``, ``C=1.0``,
  ``max_iter=2000``).
- **TF-IDF + Linear SVM (calibrated)**. The same vectoriser feeds a
  ``LinearSVC`` wrapped in ``CalibratedClassifierCV`` (3-fold sigmoid
  calibration) so that ``predict_proba`` is well-defined for calibration
  analysis later.
- **FastText**. ``fasttext.train_supervised`` with bigrams, dimension
  100, eight epochs, learning rate 0.5.

### 2.2.2 Transformer fine-tuning (Phase 4)

Four pre-trained encoders are fine-tuned with the Hugging Face
``Trainer`` over two scale tiers (small / base for the ablation
baseline, base / large for the primary configuration). All four use
``configs/models/<name>.yaml`` to keep the recipe declarative:

| Model                                  | Family       | Tokenizer         | Train batch | Precision | Recipe                                                  |
|----------------------------------------|--------------|-------------------|------------:|-----------|----------------------------------------------------------|
| ``microsoft/deberta-v3-small``         | DeBERTa-v3   | SentencePiece     | 32          | fp16      | 3 epochs, no label smoothing (ablation baseline)         |
| ``answerdotai/ModernBERT-base``        | ModernBERT   | BPE               | 16          | bf16      | 3 epochs, no label smoothing (ablation baseline)         |
| ``microsoft/deberta-v3-base``          | DeBERTa-v3   | SentencePiece     | 16          | bf16      | 5 epochs, label smoothing 0.1, cosine schedule           |
| ``answerdotai/ModernBERT-large``       | ModernBERT   | BPE               | 8           | bf16      | 5 epochs, label smoothing 0.1, cosine schedule (primary) |

All four use the Hugging Face ``Trainer`` API<sup>[30]</sup> with
linear warm-up on 10 % of the steps, AdamW<sup>[7]</sup>,
weight decay 0.01, and early stopping on validation F1-macro with
patience 2. Sequences are truncated to 256 tokens. The recipe follows
Sun et al.<sup>[31]</sup>, who systematise the cosine schedule plus
label-smoothing plus early-stopping protocol for fine-tuning BERT-family
encoders on text classification.

**Post-hoc variance-reduction ablation (Section 3.3.2.5).** The single-
seed primary run lands at F1 = 0.9498, 0.02 pp below the O1 target. As
a remedial step (not a pre-registered ablation), the primary recipe is
re-run on a 2 × 3 grid (variant ∈ {vanilla, +R-Drop} × seed ∈ {13, 42,
73}). R-Drop<sup>[9]</sup> is a consistency regulariser between
two stochastic forward passes through the same network and is
implemented as a custom ``RDropTrainer(Trainer)`` subclass that adds
the symmetric Kullback-Leibler divergence between the two output
distributions to the cross-entropy loss; it is gated by the
``training.r_drop_alpha`` field in the model YAML. The six resulting
checkpoints are then combined by soft-voting accuracy
ensembling<sup>[10]</sup> inside ``scripts/ensemble_inference.py`` to produce
the official O1 figure of F1 = 0.9528. Soft-voting deep ensembles are
also known to improve calibration and predictive
uncertainty<sup>[11]</sup>, although uncertainty is reported
separately in Section 3.5.2 via the post-hoc calibration grid rather
than as an ensemble property.

### 2.2.3 Hyper-parameter search (Phase 4.4)

After the initial fine-tune, an Optuna study over twenty trials searches
the joint space of learning rate, batch size, weight decay, warm-up ratio,
and number of epochs (see ``configs/sweeps/optuna_deberta.yaml``). The
median pruner halts trials whose intermediate validation F1 is below the
current median, halving the wall-clock cost of the sweep.

### 2.2.4 Few-shot learning (Phase 6)

SetFit<sup>[12]</sup> is trained at *N*-way *K*-shot for
*K* in {8, 16, 32, 64} with three random seeds per *K*. The body is the
``paraphrase-mpnet-base-v2`` sentence transformer; the head is a logistic
regression. The output is a learning curve that quantifies how much
labelled data is needed to recover supervised performance.

### 2.2.5 Multilingual fine-tuning (Phase 5A and 5B)

For each target language (Vietnamese, French) two encoder backbones
(**mDeBERTa-v3-base** and **XLM-R-large**) are fine-tuned in two
augmentation configurations (target-only and target + back-translation),
producing a 2 × 2 ablation grid per language and an aggregate 2 × 2 × 2
study across the two target languages:

- *Target-only*: trained on the OPUS-MT-translated training set.
- *Target + back-translation*: trained on the union of the translated
  set and its back-translation augmentation, doubling the training
  size.

All four cells per language use the same hyper-parameters as the
primary English ModernBERT-large fine-tune (5 epochs, cosine schedule,
label smoothing 0.1), except for batch size 16 on the larger
multilingual vocabularies.

## 2.3 Workflow

The project is decomposed into nine sequential phases. Each phase has a
runnable script under ``scripts/`` that reads its YAML configuration from
``configs/``, writes its artefacts under ``outputs/``, and logs metrics to
the configured experiment tracker (Weights and Biases or MLflow).

```text
Phase 1   Kickoff and Research          scripts/phase1_kickoff.py
Phase 2   Exploratory Data Analysis     scripts/phase2_eda.py
Phase 3   Baseline Models               scripts/phase3_baselines.py
Phase 4   Transformer Fine-tuning       scripts/phase4_transformers.py
Phase 5   Multilingual Extension        scripts/phase5_multilingual.py
Phase 6   Few-shot Learning             scripts/phase6_setfit.py
Phase 7   Evaluation and Explainability scripts/phase7_evaluation.py
Phase 8   Demo and Deployment           scripts/phase8_deployment.py
Phase 9   Report and Presentation       scripts/phase9_report.py
```

The dependency graph is linear: every phase consumes the artefacts of
earlier phases and produces inputs for later ones. Re-running an
intermediate phase is sufficient to refresh all downstream outputs because
each phase reads its inputs from disk rather than from a global state.

## 2.4 System Diagram

The end-to-end system is a five-layer stack. Layers are listed from
top (closest to the end user) to bottom (closest to the disk), and
each layer is materialised as a Python sub-package under ``src/`` so
the layer boundary in the diagram is also a module boundary in the
codebase. Phase scripts in ``scripts/`` orchestrate the layers without
containing business logic themselves, which keeps the library and the
entry points cleanly separated.

| # | Layer                          | Sub-package under ``src/``                                                 | Responsibilities and key artefacts                                                                                                                                                                                              |
|--:|--------------------------------|----------------------------------------------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| 5 | Deployment                     | ``src/deployment``                                                         | Multi-model Gradio app (twelve checkpoints, automatic EN/VI/FR routing, sliding-window long-document path); ONNX FP32 + dynamic INT8 export; latency-accuracy Pareto front for single-stream CPU inference.                     |
| 4 | Evaluation and Explainability  | ``src/evaluation``, ``src/explainability``                                 | Accuracy, F1-macro, confusion matrix, four-calibrator ECE grid (baseline / temperature / isotonic / chained), latency benchmarks; SHAP TextExplainer, LIME explanations, per-class and per-length error analysis.               |
| 3 | Models                         | ``src/models``, ``src/training``, ``src/inference``                        | Classical baselines (TF-IDF + Logistic Regression / Linear SVM, FastText); supervised transformers (DeBERTa-v3-small/base, ModernBERT-base/large, mDeBERTa-v3-base, XLM-RoBERTa-large); few-shot SetFit; R-Drop variant trainer. |
| 2 | Data preparation               | ``src/data``                                                               | AG News loader, stratified 90 / 10 split, Cleanlab label-noise audit, BERTopic topic discovery, OPUS-MT translation (en→vi, en→fr), back-translation augmentation, dataset versioning under ``data/processed/``.                |
| 1 | Foundation                     | ``src/configs``, ``src/utils``                                             | YAML schema (dataclass-validated), central seed control, structured logging, experiment tracking (Weights & Biases, MLflow), I/O helpers, repository-wide paths.                                                                |

The dependency direction is strictly downward: Layer N consumes
artefacts produced by Layer N - 1 and never the other way round. The
foundation layer (1) is consumed by every higher layer; the deployment
layer (5) consumes everything but is itself consumed by nothing inside
the project. This linear dependency makes it possible to re-run any
single phase (Section 2.3) and have all downstream artefacts refresh
deterministically, because each layer reads its inputs from disk
rather than from a live in-memory state.
